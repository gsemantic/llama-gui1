# Исправления для стабильной работы RAG с локальной моделью

## Проблема
При использовании RAG с локальной моделью приложение вылетало при попытке отправить запрос. 
Причина: RAG добавляет системный промпт + контекст документов (до 4500 символов ≈ 1100-1500 токенов) 
+ история разговора (до 10 сообщений ≈ 500-1000 токенов), что в сумме превышало контекстное окно модели (ctx_size = 4096 по умолчанию).

## Внесённые исправления

### 1. Уменьшен размер RAG-контекста (`src/core/rag_manager_embedding.cpp`)
- `MAX_CONTEXT_CHARS`: уменьшено с 4500 до 2500 символов (~600-700 токенов вместо ~1100-1500)
- `MAX_CHUNKS`: уменьшено с 10 до 5 чанков

**Результат:** RAG-контекст теперь занимает ~600-700 токенов вместо 1100-1500, что значительно снижает нагрузку на контекстное окно.

### 2. Добавлена защита от переполнения контекста (`src/ui/chat_message_send.cpp`)

#### Автоматический расчёт безопасного max_tokens
- Вычисляется примерный размер промпта в токенах
- Оставляется 20% запаса контекста для KV-cache и служебных токенов
- Если запрошенный max_tokens превышает доступное пространство, он автоматически уменьшается
- Минимальный max_tokens: 128 токенов (256 при критической ситуации)

#### Логирование предупреждений
- При уменьшении max_tokens выводится информация о причине
- При использовании >50% контекста выводится рекомендация увеличить ctx_size

### 3. Обработка ошибок переполнения контекста (`src/ui/chat_message_send.cpp`)

#### Обнаружение ошибок сервера
- Перехватываются JSON-ответы с ошибками от сервера
- Определяются ошибки связанные с превышением контекста:
  - "context size exceeded"
  - "n_ctx" errors
  - "exceed" errors
  - "too long" errors

#### Информативное сообщение об ошибке
При обнаружении переполнения контекста пользователь получает:
- Текущий размер контекста
- Примерный размер промпта
- Конкретные рекомендации по исправлению:
  1. Увеличить ctx_size (минимум 8192 для RAG)
  2. Уменьшить длину сообщений
  3. Очистить историю чата
  4. Уменьшить количество документов в RAG

## Рекомендации по настройкам

### Для работы с RAG на слабом оборудовании

**Минимальные настройки:**
```
ctx_size = 8192          # Минимум для RAG (было 4096)
max_tokens = 512         # Уменьшено для экономии контекста
max_chunks_in_memory = 200  # Уменьшено с 500
max_rag_chunks = 5       # Уже изменено в коде
```

**Оптимальные настройки:**
```
ctx_size = 16384         # Для комфортной работы с RAG
max_tokens = 1024        # Стандартное значение
max_chunks_in_memory = 300
max_rag_chunks = 7
```

### Влияние на качество
- Уменьшение чанков с 10 до 5 незначительно снижает полноту контекста
- Однако для большинства запросов 5 наиболее релевантных чанков достаточно
- При необходимости можно увеличить `similarity_threshold` для более точного отбора

## Тестирование

Проверьте работу RAG с локальной моделью:
1. Загрузите документы в RAG
2. Отправьте несколько запросов
3. Проверьте логи на наличие сообщений `[RAG PROTECTION]` и `[RAG WARNING]`
4. Если видите предупреждения - увеличьте ctx_size в настройках сервера

## Технические детали

### Расчёт токенов
- Используется грубая оценка: 1 токен ≈ 4 символа (для русского/английского)
- Для более точного подсчёта можно использовать токенизатор модели

### Запас контекста
- 20% контекста оставляется для KV-cache и служебных токенов
- Это предотвращает переполнение при генерации ответа

### Совместимость
- Изменения обратно совместимы
- При отключении RAG поведение не меняется
- Облачные модели не затронуты (у них большие контекстные окна)
